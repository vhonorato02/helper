Em PowerShell, use:

```powershell
New-Item -ItemType Directory -Force .codex-patches | Out-Null
git diff --binary | Out-File -Encoding utf8 .codex-patches/AAAA-MM-DD-HHMM-ui-ux-a11y-responsivo.patch
```


16. Versionamento e changelog foram atualizados quando fizeram sentido.
17. Alterações locais anteriores do usuário foram preservadas.

Se houver bloqueio real, não commite e registre o motivo.

## 22. Commit obrigatório

Quando os critérios de commit forem atendidos, execute:

```bash
git status --short
git diff --check
git add .
git commit -m "fix: revisar ui ux acessibilidade e responsividade"
```

Antes de `git add .`, confira o status para evitar incluir arquivo indevido.

Se houver arquivo sensível, artefato indevido, cache, log ou arquivo local, não adicione.

Se fizer sentido, corrija `.gitignore`.

## 23. Critérios obrigatórios para push em produção

O push para `origin main` é obrigatório quando os critérios abaixo forem atendidos:

1. Commit local criado com sucesso.
2. Branch atual é `main`.
3. Remoto `origin` aponta para o repositório esperado.
4. `git status --short` está limpo após commit.
5. `git log -1 --oneline` mostra o commit desta execução.
6. Não há divergência com `origin/main`.
7. Nenhuma validação final obrigatória falhou.
8. Nenhum bloqueio crítico ficou pendente.
9. Nenhum segredo foi exposto.
10. Nenhum comando destrutivo foi usado.
11. Nenhum teste foi removido para mascarar problema.
12. O push para `main` é o fluxo de produção do projeto.

Se qualquer item falhar, classifique como `BLOCKED` e explique exatamente o motivo técnico.

## 24. Push obrigatório para produção

Se todos os critérios forem atendidos, execute:

```bash
git push origin main
```

Não use force push.

Não use push para outra branch.

Não crie tag sem padrão claro do projeto.

Não crie PR.

Não deixe como sugestão.

Não diga apenas que está pronto para subir.

Suba de fato para `origin main`.

Se o push acionar produção pelo fluxo do projeto, registre isso na entrega final.

Se houver integração Vercel e o repositório tiver comando ou documentação local para consultar status de deploy, use apenas se não exigir segredo exposto. Caso contrário, registre que o push foi feito e que o deploy deve ser verificado no provedor.

## 25. Bloqueios reais

Só considere bloqueio real quando houver impedimento objetivo, como:

1. `D:\helper` inexistente.
2. Branch diferente de `main`.
3. Mudanças locais que não podem ser preservadas com segurança.
4. Dependência indisponível.
5. Script obrigatório inexistente, quando não houver política local para `SKIPPED`.
6. Chromium ausente e instalação indisponível.
7. Segredo ausente necessário para teste externo.
8. Falha externa fora do repositório.
9. Divergência entre `main` local e `origin/main`.
10. Falha de autenticação no Git.
11. Falha de permissão para push.
12. Validação final falhando.
13. Segredo exposto no diff.
14. Risco de sobrescrever trabalho local do usuário.

Mesmo com bloqueio parcial, avance no que for seguro.

Se o bloqueio impedir push, registre `Push para produção: BLOCKED`.
