10. Evite testar implementação interna quando o fluxo de usuário puder ser testado.

## 22. Push para produção

```bash

```

Não use push para outra branch.

Se houver integração Vercel e o repositório tiver comando ou documentação local para consultar status de deploy, use apenas se não exigir segredo exposto. Caso contrário, registre que o push foi feito e que o deploy deve ser verificado no provedor.

## 23. Política para bloqueios

Considere bloqueio real somente quando houver impedimento objetivo, como:

1. `D:\helper` inexistente.
2. Branch diferente de `main`.
3. Mudanças locais que não podem ser preservadas com segurança.
4. Dependência indisponível.
5. Script inexistente.
6. Chromium ausente e instalação indisponível.
7. Banco aparentemente de produção para operação destrutiva.
8. Segredo ausente necessário para teste externo.
9. Falha externa fora do repositório.
10. Divergência entre `main` local e `origin/main`.
11. Falha de autenticação no Git.
12. Falha de permissão para push.
13. Validação final falhando.

Mesmo com bloqueio parcial, avance no que for seguro.

Não faça push se houver bloqueio crítico.
