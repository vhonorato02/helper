# PROMPT DE EXECUÇÃO AUTÔNOMA: REPARO PROFUNDO DIÁRIO DO HELPER

Você é um agente sênior de manutenção full stack. Sua tarefa é executar correções reais no repositório Helper, diretamente no working tree local, com validação completa, patch auditável e entrega objetiva.

Este prompt representa uma nova execução de patch. Cada execução deve produzir alterações rastreáveis, ou registrar claramente por que não houve alteração possível.

## 1. Regras absolutas de ambiente

Cumpra todas as regras abaixo sem exceção operacional.

1. Trabalhe somente em:

```powershell
D:\helper
```

2. Trabalhe diretamente na branch:

```text
main
```

3. Não crie branch.

4. Não crie worktree.

5. Não use workspace temporário.

6. Não copie o projeto para outro diretório.

7. Não use `npm`.

8. Use somente `pnpm`.

9. Faça todas as alterações diretamente no working tree real de `D:\helper`.

10. Não esconda alteração fora do Git.

11. Não apague trabalho local do usuário.

12. Não use comandos destrutivos como:

```bash
git reset --hard
git clean -fd
git clean -fdx
rm -rf
```

13. Não remova testes para fazer a validação passar.

14. Não desative regras de lint para ocultar erro.

15. Não esconda erro com `any`, `@ts-ignore`, `@ts-expect-error`, casts inseguros ou supressões artificiais.

16. Não imprima segredos, tokens, senhas, cookies, connection strings ou variáveis sensíveis.

17. Se um comando exibir segredo acidentalmente, interrompa a exposição, redija o valor na resposta e prossiga apenas com segurança.

18. Se `D:\helper` não existir, não crie outro diretório. Registre bloqueio.

19. Se a branch atual não for `main`, não troque automaticamente se houver risco de perda de trabalho. Registre bloqueio e explique.

20. Se houver mudanças locais antes da execução, preserve tudo. Nunca sobrescreva trabalho existente.

## 10. Critérios de correção

Corrija problemas com foco em causa raiz.

Não faça correção cosmética se houver falha funcional, teste quebrado, erro de tipo ou risco de segurança pendente.

### 10.1 TypeScript

1. Elimine `any` quando houver tipo inferível ou modelável.
2. Não use `as unknown as` salvo quando houver fronteira externa inevitável e validação explícita.
3. Não use `@ts-ignore`.
4. Não use `@ts-expect-error` para esconder problema permanente.
5. Modele payloads com Zod quando vierem de request, formulário, URL, cookie, banco ou API externa.
6. Preserve tipos de Server Actions, Route Handlers e componentes client/server.

### 10.2 Next.js App Router

1. Respeite fronteira entre Server Components e Client Components.
2. Não use `window`, `document`, `localStorage` ou `sessionStorage` em Server Component.
3. Quando uso client-side for necessário, proteja com componente client e checagem de ambiente.
4. Corrija metadata, redirects, cookies e cache conforme padrão existente.
5. Não transforme página server em client sem necessidade.

### 10.3 Autenticação e segurança

1. Preserve cookie HTTP-only.
2. Preserve uso seguro de JWT com `jose`.
3. Valide entrada com Zod.
4. Não exponha segredo em client bundle.
5. Não coloque variável sensível em arquivo público.
6. Não registre segredo em log.
7. Não relaxe proteção de rota.
8. Não desabilite autenticação para fazer teste passar.
9. Corrija mensagens de erro para não revelar detalhes internos.
10. Verifique rotas públicas de solicitação com atenção especial contra entrada inválida.

### 10.4 Banco, Drizzle e Neon

1. Não execute `db:setup` se `DATABASE_URL` aparentar ser produção.
2. Não faça migração destrutiva.
3. Não apague tabela, coluna ou dado sem instrução explícita.
4. Se alterar schema, atualize arquivos de schema relacionados.
5. Se houver migração existente no padrão do projeto, siga esse padrão.
6. Se não for seguro migrar, registre pendência bloqueada.
7. Valide coerência entre `src/db/schema.ts` e `database/schema.sql` quando ambos existirem.

### 10.5 UI, estados e mobile

Corrija problemas reais de:

1. Loading state ausente.
2. Empty state ausente.
3. Error state ausente.
4. Feedback de ação ausente.
5. Layout quebrado em mobile.
6. Botão sem estado disabled durante submissão.
7. Formulário sem validação clara.
8. Falta de acessibilidade básica.
9. Contraste ruim se estiver evidente no código.
10. Navegação inconsistente.

### 10.6 PWA

Verifique e corrija quando necessário:

1. Manifest.
2. Ícones.
3. Service worker, se existir.
4. Metadata.
5. Instalação.
6. Compatibilidade com build.
7. Rotas públicas e cache.

### 10.7 Testes

1. Corrija testes quebrados pela causa raiz.
2. Adicione ou ajuste testes quando comportamento mudar.
3. Não remova teste para passar validação.
4. Não transforme teste em trivialidade sem assertiva útil.
5. Cubra pelo menos o fluxo alterado quando houver estrutura de teste existente.
6. Se teste e2e depender de ambiente indisponível, registre bloqueio com evidência objetiva.

## 13. Patch obrigatório em `.codex-patches`

Cada execução precisa gerar patch.

Antes de finalizar, crie a pasta se ela não existir:

```bash
mkdir -p .codex-patches
```

Gere um patch com nome baseado em data, hora e resumo curto.

Exemplo:

```bash
git diff --binary > .codex-patches/2026-06-24-reparo-profundo-helper.patch
```

Se o ambiente for PowerShell puro, use alternativa compatível:

```powershell
New-Item -ItemType Directory -Force .codex-patches | Out-Null
git diff --binary | Out-File -Encoding utf8 .codex-patches/2026-06-24-reparo-profundo-helper.patch
```

O patch deve incluir todas as alterações da execução.

Depois de gerar o patch, rode:

```bash
git status --short
```

Não deixe alteração relevante fora do Git sem mencionar.

## 15. Política para bloqueios

Só considere bloqueio quando houver impedimento real, como:

1. falta de acesso a `D:\helper`;
2. branch incorreta com risco de sobrescrever trabalho;
3. dependência indisponível;
4. segredo ausente necessário para teste externo;
5. banco aparentemente de produção;
6. script inexistente;
7. falha externa fora do repositório;
8. permissão negada.

Mesmo com bloqueio parcial, avance no que for seguro.

Não pare a execução inteira se houver correções locais possíveis.

6. remoção de teste;
7. desativação de lint;
8. exposição de segredo;
9. criação de branch;
10. criação de worktree;
11. uso de `npm`;
12. alteração fora de `D:\helper`;
13. limpeza destrutiva do working tree;
14. commit automático sem pedido explícito;
15. push automático sem pedido explícito;
16. deploy automático sem pedido explícito.

## 18. Condição de sucesso

A execução é considerada bem-sucedida quando:

1. correções reais foram aplicadas;
2. alterações estão no working tree real de `D:\helper`;
3. patch foi criado em `.codex-patches`;
4. validações finais foram executadas;
5. falhas restantes, se houver, estão claramente justificadas como bloqueio;
6. nenhum segredo foi exposto;
7. nenhum trabalho local foi apagado;
8. nenhum teste foi removido para mascarar problema;
9. a entrega final mostra exatamente o que mudou.
